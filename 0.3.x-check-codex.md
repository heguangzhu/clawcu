# ClawCU 0.3.x Command Surface Check

Date: 2026-04-29
Reviewer: Codex
Branch reviewed: `main`

This document reviews whether each visible ClawCU command and major option group is necessary for the 0.3.x product surface. The goal is not to shrink the CLI for its own sake. The goal is to keep the command surface clear, durable, and explainable before publishing a stable 0.3.x release.

## Review Inputs

- Source entrypoint checked with `PYTHONPATH=src python3 -m clawcu --help`.
- Current globally installed `clawcu` is still `0.2.12`, while source `main` exposes 0.3.x A2A commands.
- Reviewed command definitions in:
  - `src/clawcu/cli.py`
  - `src/clawcu/a2a/cli.py`
  - `src/clawcu/core/service.py`
  - `src/clawcu/dashboard/server.py`
  - `src/clawcu/dashboard/actions.py`
- Included known review findings from the 0.3.x hardening pass.

## Decision Legend

- `Keep`: core to the product promise and should remain user-facing.
- `Keep, refine`: necessary, but needs UX, safety, naming, or docs work.
- `Advanced`: useful, but should be framed as advanced, experimental, or operator-oriented.
- `Candidate to hide/deprecate`: not clearly needed as a primary command; consider hiding from common docs, merging, renaming, or deprecating.

## Executive Summary

ClawCU's command surface has a strong core: lifecycle management for OpenClaw and Hermes, safe upgrades and rollbacks, environment/provider handling, and operational visibility. Those are necessary.

The 0.3.x additions create two areas of extra complexity:

- A2A introduces a second networking model with registry, bridge, card, send, sidecar baking, and hop-budget controls.
- The dashboard introduces a browser control plane with powerful actions, but currently has authentication and platform-portability concerns.

The recommended direction is:

- Keep the shared lifecycle commands.
- Keep provider workflows, but tighten naming and reduce legacy option noise.
- Keep A2A as a 0.3.x feature, but mark fallback bridge/registry commands as advanced or dev-oriented.
- Keep dashboard, but block or protect non-loopback serving before calling it stable.
- Consider hiding redundant aliases and alternate forms from primary docs, even if they remain for compatibility.

## Release Blockers

These should be fixed before presenting 0.3.x as stable.

| Area | Problem | Recommendation |
| --- | --- | --- |
| Dashboard remote exposure | `clawcu dashboard --host 0.0.0.0` can expose inspect, logs, token retrieval, clone, upgrade helpers, and rollback over unauthenticated HTTP. | Keep dashboard loopback-only by default and require an explicit auth token or an explicit unsafe flag for non-loopback binds. |
| Dashboard platform actions | Dashboard action buttons shell out through `osascript`, so Linux support is overstated for those actions. | Add Linux-compatible launch behavior or return a clear unsupported-platform error in the UI/API. |
| Forward compatibility | `InstanceRecord.from_dict()` and `ProviderRecord.from_dict()` drop unknown keys; later save paths can erase future metadata. | Preserve unknown fields or avoid rewriting records in read-only flows. |
| `list --no-cache` cache loss | A partial refresh can discard still-valid cache entries for another service. | Start from the existing cache even when bypassing read hits, then only replace successful service entries. |
| Custom datadir orphan lifecycle | `list --removed` only scans direct children of `CLAWCU_HOME`; removed instances with external datadirs become undiscoverable. | Store a tombstone/index entry for kept datadirs or maintain a removed-instance registry. |
| Version metadata mismatch | Source `main` exposes A2A commands, but `python -m clawcu --version` reports `0.2.12`. | Bump package metadata before release and verify global install shows the intended 0.3.x version. |

## Redundancy and Naming Notes

| Item | Concern | Recommendation |
| --- | --- | --- |
| `create --service` vs `create openclaw/hermes` | Two ways to do the same thing. | Use service subcommands in docs; keep `--service` for compatibility only. |
| `pull --service` vs `pull openclaw/hermes` | Same concern. | Prefer subcommands in docs. |
| `list --source removed` vs `list --removed` | Useful shortcut, but more flags to explain. | Keep both, make `--source` canonical. |
| `remove --removed` | Semantically odd but practical. | Keep, but docs should say "delete leftover datadir" rather than "remove instance". |
| `config` as shared command | Same command maps to different native flows. | Keep, but explicitly document service mapping and expected behavior. |
| `tui` as shared command | Same command launches different service-native UI/chat flows. | Keep, but clarify service differences. |
| `dashboard` and `token` | Both can expose secrets. | Keep, but treat token visibility and remote dashboard as security-sensitive. |

## Suggested 0.3.x Command Documentation Shape

The docs should present commands in layers:

1. First-run core:
   - `setup`
   - `create openclaw`
   - `create hermes`
   - `list`
   - `inspect`
   - `start` / `stop`
   - `logs`
   - `tui`

2. Safe lifecycle:
   - `clone`
   - `upgrade --dry-run`
   - `upgrade`
   - `rollback --list`
   - `rollback`
   - `recreate`
   - `remove`

3. Configuration:
   - `config`
   - `setenv` / `getenv` / `unsetenv`
   - `provider collect/list/show/apply/remove/models`
   - `hermes identity set`

4. Advanced networking:
   - `create --a2a`
   - `recreate --a2a/--no-a2a`
   - `a2a card`
   - `a2a send`
   - `a2a registry serve`
   - `a2a up`
   - `a2a bridge serve`

5. Visual operations:
   - `dashboard`

This order keeps daily users away from A2A/demo/networking detail until they need it.

## Bottom Line

Most commands are justified. The product has earned its breadth because it manages real local runtimes, versions, env files, providers, Docker state, and safety snapshots.

The main issue is not that there are too many commands. The issue is that 0.3.x now has multiple maturity levels sharing the same command surface:

- stable lifecycle commands;
- service-specific convenience commands;
- provider migration commands;
- dashboard control-plane commands;
- A2A experimental/operator commands.

For a stable release, ClawCU should keep the breadth but make maturity and risk explicit. The core lifecycle should feel boring and reliable. A2A and dashboard should feel powerful, but clearly bounded.
