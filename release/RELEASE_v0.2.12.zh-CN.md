# ClawCU v0.2.12

🌐 Language:
[English](RELEASE_v0.2.12.md) | [中文](RELEASE_v0.2.12.zh-CN.md)

发布日期：2026 年 4 月 22 日

> `v0.2.12` 继续打磨日常最常跑的 `list` 路径：现在可以用 `--no-cache` 主动绕过当天的 Available Versions 缓存，立刻向 registry 取最新结果；同时，历史实例记录里残留的旧字段也不再把 `clawcu list` 直接打崩。

* * *
## 亮点

- **`clawcu list --no-cache`**
  - 强制为页脚的 Available Versions 重新拉一遍远端 tag。
  - 不影响主实例表的行为。
  - 这次成功拉到的结果仍会回写到本地缓存，所以后续普通 `list` 继续是热缓存。

- **历史记录兼容性**
  - 较老的实例 JSON 里可能会带一些新版本代码已经不再显式建模的字段。
  - 现在 `InstanceRecord.from_dict()` 会忽略未知字段，而不是直接报错退出。
  - 现实效果就是：`clawcu list`、`inspect` 这些命令面对旧状态文件时更稳，不需要先手工清理记录。

- **文档口径对齐**
  - README 与 usage 文档里，`config` 的映射已经改成与实现一致：OpenClaw 对应 `configure`，Hermes 对应 `setup`。
  - `list` 的文档也补上了显式刷新缓存的路径。

- **发布时测试数 396 条**，其中新增覆盖 `list --no-cache` 和旧实例记录兼容加载。

* * *
## 为什么要有 `--no-cache`

`v0.2.10` 给 `clawcu list` 加上了按天缓存，这是完全正确的默认值：快、稳，而且不会每次都打 registry。

但缓存一旦存在，就还缺一个很实际的运维动作：

"我知道远端已经更新了，这一次就给我看最新的。"

`v0.2.12` 补上的就是这条路径：

```bash
clawcu list --no-cache
```

它不是永久关闭缓存，而是只让这次调用跳过"读缓存"这一步；如果拉取成功，仍然会把新结果回写到缓存文件里。所以新的 UX 是：

- 默认快；
- 需要时可强制刷新；
- 刷新完之后继续保持热缓存。

* * *
## 为什么旧记录兼容也很重要

在真实命令验证里，我们碰到过一个不应该由用户承担的问题：历史实例记录里残留了像 `a2a_enabled` 这样的旧字段，而新代码已经不再显式建模它们，结果 `clawcu list` 在反序列化阶段就直接失败。

`v0.2.12` 把规则改成了：

- 已知字段正常加载；
- 未知历史字段直接忽略。

这对本地生命周期状态来说是更合理的契约：

- 老记录应该尽量可读；
- schema 清理不应该把用户留在半路；
- CLI 面对历史包袱时应优雅退化，而不是先崩掉。

* * *
## 兼容性

`v0.2.12` 是 `v0.2.11` 的无痛升级。

- 没有破坏性 CLI 变更。
- `clawcu list` 默认仍然走缓存路径。
- `--no-cache` 是纯新增、可选项。
- 旧实例记录里带额外字段的情况，现在更容易直接继续工作，不需要手动清理。

* * *
## 推荐工作流

只想快速看当前全局状态时，继续用默认路径：

```bash
clawcu list
```

明确需要最新 registry 结果时，再显式刷新：

```bash
clawcu list --no-cache
```

然后继续正常的生命周期动作：

```bash
clawcu inspect writer
clawcu upgrade writer --version 2026.4.21
```

* * *
## 结语

`v0.2.11` 让 runtime image 覆盖成为生命周期中的耐久状态；`v0.2.12` 虽然更小，但延续的是同一种产品取向：默认路径继续快，显式路径继续可控，而历史本地状态也应该被更稳妥地对待。

接下来仍然会往 `0.3.0` 推进：统一的 `--output {table|json|yaml}` 协议、provider bundle 溯源，以及把 active-provider 升为一等字段。
