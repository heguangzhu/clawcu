# ClawCU 0.3.x 命令面检查

日期：2026-04-29
评审者：Codex
评审分支：`main`

本文档评审 ClawCU 当前可见命令以及主要参数组在 0.3.x 产品面里是否有必要保留。目标不是为了缩小 CLI 而缩小 CLI，而是在发布稳定的 0.3.x 之前，让命令面保持清晰、耐用、可解释。

## 评审输入

- 使用 `PYTHONPATH=src python3 -m clawcu --help` 检查源码入口。
- 当前全局安装的 `clawcu` 仍是 `0.2.12`，但源码 `main` 已经暴露 0.3.x 的 A2A 命令。
- 检查了这些命令定义：
  - `src/clawcu/cli.py`
  - `src/clawcu/a2a/cli.py`
  - `src/clawcu/core/service.py`
  - `src/clawcu/dashboard/server.py`
  - `src/clawcu/dashboard/actions.py`
- 纳入了 0.3.x 稳定性加固评审中已发现的问题。

## 判断标准

- `保留`：属于产品承诺的核心能力，应该继续作为用户可见功能存在。
- `保留，但需调整`：功能有必要，但需要 UX、安全、命名或文档层面的修正。
- `高级功能`：有价值，但应该被标记为高级、实验性或运维人员使用场景。
- `可隐藏/可弃用候选`：不一定适合作为主命令面暴露；可以考虑从常用文档中隐藏、合并、改名或逐步弃用。

## 总结

ClawCU 的核心命令面是成立的：OpenClaw 与 Hermes 的生命周期管理、安全升级与回滚、环境变量和 provider 管理、运行状态可观测性。这些能力都有必要保留。

0.3.x 的新增能力主要带来了两类复杂度：

- A2A 引入了第二套网络模型，包括 registry、bridge、card、send、sidecar baking、hop budget 等概念。
- dashboard 引入了浏览器控制台能力，功能很强，但当前存在鉴权和跨平台行为问题。

建议方向：

- 保留共享生命周期命令。
- 保留 provider 工作流，但收紧命名和遗留参数噪音。
- 保留 A2A 作为 0.3.x 功能，但把 fallback bridge / registry 命令标记为高级或开发者向。
- 保留 dashboard，但在称为稳定功能之前，必须阻断或保护非 loopback 访问。
- 冗余别名和替代形式可以为了兼容继续存在，但不应放在主文档路径里重点推荐。

## 发布阻塞项

这些问题应该在把 0.3.x 称为稳定版本之前修复。

| 范围 | 问题 | 建议 |
| --- | --- | --- |
| Dashboard 远程暴露 | `clawcu dashboard --host 0.0.0.0` 可能把 inspect、logs、token 获取、clone、upgrade 辅助和 rollback 通过未鉴权 HTTP 暴露到网络上。 | 默认只允许 loopback；如果绑定非 loopback，需要显式 auth token 或显式 unsafe 参数。 |
| Dashboard 平台行为 | Dashboard 的 action 按钮依赖 `osascript`，因此 Linux 支持被高估了。 | 增加 Linux 兼容启动方式，或者在 UI/API 中返回清晰的“不支持此平台”。 |
| 前向兼容 | `InstanceRecord.from_dict()` 和 `ProviderRecord.from_dict()` 会丢弃未知字段；后续保存路径可能擦除未来版本写入的元数据。 | 保留未知字段，或者避免只读流程重写加载后的记录。 |
| `list --no-cache` 缓存丢失 | 部分刷新成功、部分刷新失败时，可能丢掉另一个服务仍然有效的缓存项。 | 即使绕过缓存命中，也应从现有缓存开始，只替换成功刷新的服务项。 |
| 自定义 datadir 的 orphan 生命周期 | `list --removed` 只扫描 `CLAWCU_HOME` 的直接子目录；外部 datadir 的 removed 实例会不可发现。 | 为保留数据目录的 remove 写 tombstone/index，或维护 removed-instance registry。 |
| 版本元数据不一致 | 源码 `main` 暴露 A2A 命令，但 `python -m clawcu --version` 仍显示 `0.2.12`。 | 发布前 bump package metadata，并确认全局安装后显示目标 0.3.x 版本。 |

## 建议的 0.3.x 文档组织

文档应该按层次呈现命令：

1. 首次使用核心路径：
   - `setup`
   - `create openclaw`
   - `create hermes`
   - `list`
   - `inspect`
   - `start` / `stop`
   - `logs`
   - `tui`

2. 安全生命周期：
   - `clone`
   - `upgrade --dry-run`
   - `upgrade`
   - `rollback --list`
   - `rollback`
   - `recreate`
   - `remove`

3. 配置：
   - `config`
   - `setenv` / `getenv` / `unsetenv`
   - `provider collect/list/show/apply/remove/models`
   - `hermes identity set`

4. 高级网络：
   - `create --a2a`
   - `recreate --a2a/--no-a2a`
   - `a2a card`
   - `a2a send`
   - `a2a registry serve`
   - `a2a up`
   - `a2a bridge serve`

5. 可视化运维：
   - `dashboard`

这个顺序可以让日常用户在需要之前不必理解 A2A、demo bridge 和网络细节。

## 结论

大多数命令都有必要保留。ClawCU 的命令面之所以变宽，是因为它确实在管理真实的本地 runtime、版本、env 文件、provider、Docker 状态和安全快照。

主要问题不是“命令太多”，而是 0.3.x 现在把多个成熟度层级放在同一个命令面里：

- 稳定生命周期命令；
- 服务专属便利命令；
- provider 迁移命令；
- dashboard 控制面命令；
- A2A 实验性/运维命令。

稳定版本应该保留这些宽度，但要把成熟度和风险讲清楚。核心生命周期应该足够朴素可靠；A2A 和 dashboard 可以很强，但边界必须清楚。
